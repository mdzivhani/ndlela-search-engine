# Documentation Style Guide## PurposeGood documentation makes code maintainable, helps onboarding, and serves as a reference. This guide covers how to write effective documentation for code, APIs, and projects.## Principles- **Clarity**: Write for your audience's understanding level- **Conciseness**: Be brief but complete- **Accuracy**: Keep documentation in sync with code- **Searchability**: Use clear headings and structure- **Examples**: Show, don't just tell- **Maintenance**: Update docs when code changes## MarkdownUse Markdown for all documentation files:- README.md files- Architecture Decision Records (ADRs)- API documentation- Development guides- Release notes### Markdown Best Practices\# Main Title (H1) - Only one per documentBrief introduction paragraph.\## Section (H2)\### Subsection (H3)Regular paragraph text.\- Bullet point\- Another bullet point\- Nested bullet1\.
