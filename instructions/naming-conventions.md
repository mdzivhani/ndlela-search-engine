# Naming ConventionsConsistent naming makes code more readable and maintainable. Follow these conventions across all projects.## General Principles- **Be Descriptive**: Names should clearly convey purpose- **Be Consistent**: Use the same patterns throughout the codebase- **Be Concise**: Avoid unnecessary verbosity while maintaining clarity- **Avoid Abbreviations**: Unless universally understood (ID, HTTP, URL)- **Use Domain Language**: Use terminology familiar to the business domain## Backend (C# / .NET)### Namespaces- Use PascalCase- Follow folder structure- Pattern: CompanyName.ProductName.Featurenamespace Company.FleetManager.Services;namespace Company.Core.Common.Extensions;### Classes- Use PascalCase (UpperCamelCase)- Use nouns or noun phrases- Be specific and descriptivepublic class UserService { }public class DataTransformer { }public class CustomerRepository { }public class OrderValidator { }### Interfaces- Use PascalCase with I prefix- Use adjectives or nounspublic interface IUserService { }public interface IDataTransformer { }public interface IDisposable { }public interface IComparable { }### Methods- Use PascalCase- Start with verb- Be specific about what the method doespublic User GetUserById(Guid id) { }public async Task&lt;Order&gt; CreateOrderAsync(OrderDto dto) { }public bool ValidateInput(string input) { }public void ProcessPayment(Payment payment) { }#### Async Methods- Always suffix with Asyncpublic async Task&lt;User&gt; GetUserAsync(Guid id) { }public async Task SaveChangesAsync() { }#### Boolean Methods- Use Is, Has, Can, Shouldpublic bool IsValid() { }public bool HasPermission() { }public bool CanExecute() { }public bool ShouldRetry() { }### Properties- Use PascalCase- Use nouns or noun phrases- Boolean properties use Is, Has, Canpublic string UserName { get; set; }public int OrderCount { get; set; }public bool IsActive { get; set; }public bool HasChildren { get; set; }public bool CanEdit { get; set; }### Fields#### Private Fields- Use camelCase with _ prefixprivate readonly ILogger \_logger;private int \_retryCount;private const int \_maxRetries = 3;#### Public/Protected Fields (rare)- Use PascalCasepublic readonly string Version = "1.0";### Parameters & Local Variables- Use camelCase- Be descriptivepublic void ProcessOrder(Order order, int quantity, decimal price){var totalPrice = price \* quantity;var discountedPrice = ApplyDiscount(totalPrice);}### Constants- Use PascalCase for private constants- Use UPPER_SNAKE_CASE only when following external conventionsprivate const int MaxRetryCount = 3;private const string DefaultConnectionString = "...";public const string API_VERSION = "v1"; // External API convention### Enums- Use PascalCase for enum name and values- Use singular for enums (unless bit flags)public enum OrderStatus{Pending,Processing,Completed,Cancelled}\[Flags\]public enum FileAccess{Read = 1,Write = 2,Execute = 4}### Generic Type Parameters- Use T for single type parameter- Use descriptive names with T prefix for multiple parameterspublic class Repository&lt;T&gt; { }public class Mapper&lt;TSource, TDestination&gt; { }public class Cache&lt;TKey, TValue&gt; { }### Events- Use PascalCase- Use present or past tense verbpublic event EventHandler UserLoggedIn;public event EventHandler OrderProcessing;public event EventHandler&lt;OrderEventArgs&gt; OrderCompleted;### DTOs (Data Transfer Objects)- Suffix with Dtopublic class UserDto { }public class CreateOrderDto { }public class UpdateVehicleDto { }### Test Classes- Suffix with Tests- Match the class being testedpublic class UserServiceTests { }public class DataTransformerTests { }## Frontend (TypeScript / Angular)### Variables & Functions- Use camelCase- Be descriptiveconst userName = 'John';let orderCount = 0;function calculateTotal(items: Item\[\]): number { }### Constants- Use UPPER_SNAKE_CASE for true constants- Use camelCase for configuration objectsconst MAX_RETRY_COUNT = 3;const API_BASE_URL = '<https://api.example.com>';const appConfig = {retryCount: 3,timeout: 5000};### Classes & Interfaces- Use PascalCase- Don't prefix interfaces with Iclass UserService { }class ProductComponent { }interface User { }interface ProductState { }### Enums- Use PascalCase for enum name and valuesenum OrderStatus {Pending = 'PENDING',Processing = 'PROCESSING',Completed = 'COMPLETED'}### Type Aliases- Use PascalCasetype UserId = string;type Callback = () => void;type HttpHeaders = Record&lt;string, string&gt;;### Boolean Variables- Use is, has, can, shouldconst isLoading = true;const hasPermission = false;const canEdit = true;const shouldRetry = false;### Functions/Methods- Use camelCase- Start with verb- Be specificfunction getUserById(id: string): User { }function loadUsers(): void { }async function saveChanges(): Promise&lt;void&gt; { }### Observables (RxJS)- Suffix with \$const users\$ = this.http.get&lt;User\[\]&gt;('/api/users');const isLoading\$ = this.store.select(selectIsLoading);### Angular Components- Suffix with Component@Component({...})export class UserListComponent { }@Component({...})export class ProductDetailComponent { }### Angular Services- Suffix with Service@Injectable()export class UserService { }@Injectable()export class AuthenticationService { }### Angular Directives- Suffix with Directive- Use camelCase selector with prefix@Directive({selector: '\[appHighlight\]'})export class HighlightDirective { }### Angular Pipes- Suffix with Pipe- Use camelCase name@Pipe({ name: 'formatDate' })export class FormatDatePipe { }### ngrx Actions- Use square brackets for source- Use present tense verbexport const loadUsers = createAction('\[Users Page\] Load Users');export const loadUsersSuccess = createAction('\[Users API\] Load Users Success',props&lt;{ users: User\[\] }&gt;());export const loadUsersFailure = createAction('\[Users API\] Load Users Failure',props&lt;{ error: string }&gt;());### ngrx Reducers- Use camelCase with Reducer suffixexport const usersReducer = createReducer(initialState, ...);### ngrx Selectors- Prefix with selectexport const selectUsers = createSelector(...);export const selectIsLoading = createSelector(...);export const selectUserById = (id: string) => createSelector(...);### ngrx Effects- Suffix with \$loadUsers\$ = createEffect(() => ...);createUser\$ = createEffect(() => ...);## File Naming### Backend (C#)- Match the primary type name- Use PascalCaseUserService.csIUserRepository.csOrderDto.csPathTransformerTests.cs### Frontend (TypeScript/Angular)- Use kebab-case- Include type suffixuser.service.tsuser.service.spec.tsuser-list.component.tsuser-list.component.htmluser-list.component.scssuser.model.tsuser.actions.tsuser.reducer.tsuser.effects.ts### Configuration Files- Use kebab-case- Be descriptiveangular.jsontsconfig.json.editorconfig.eslintrc.jsondocker-compose.yml## Folder Naming### Backend- Use PascalCaseServices/Controllers/Repositories/Models/Extensions/### Frontend- Use kebab-casecomponents/services/models/store/shared/features/## Database Naming### Tables- Use PascalCase singular- Be specificUserProductOrderTransaction### Columns- Use PascalCase- Match C# property namesUserIdFirstNameCreatedAtIsActive### Foreign Keys- Suffix with IdUserIdProductIdOrderId### Indexes- Prefix with IX_- Include table and column namesIX_Users_EmailIX_Orders_UserIdIX_Products_Category_CreatedAt## URL & Endpoint Naming### REST APIs- Use kebab-case- Use plural nouns for collections- Use HTTP verbs for actionsGET /api/usersGET /api/users/{id}POST /api/usersPUT /api/users/{id}DELETE /api/users/{id}GET /api/users/{id}/ordersPOST /api/products/{id}/commands/activate### Frontend Routes- Use kebab-case- Be descriptive/dashboard/users/users/:id/users/:id/edit/products/inventory/settings/notifications## Environment Variables- Use UPPER_SNAKE_CASEDATABASE_CONNECTION_STRINGAPI_BASE_URLREDIS_CONNECTION_STRINGENABLE_FEATURE_X## CSS/SCSS### Classes- Use kebab-case- Follow BEM or similar methodology.user-list { }.user-list_\_item { }.user-list_\_item--active { }.btn { }.btn-primary { }.btn-large { }### IDs- Use kebab-case (rare in Angular)# main-navigation { }# user-profile { }## Git### Branch Names- Use kebab-case- Include type prefixfeature/add-user-authenticationbugfix/fix-payment-calculationhotfix/critical-security-issuerefactor/improve-error-handling### Commit Messages- Use imperative mood- Start with capital letter- No period at endAdd user authenticationFix payment calculation bugRefactor error handling logicUpdate dependencies## Anti-Patterns to Avoid### Don't Use Hungarian Notation// Badstring strUserName;int iCount;// Goodstring userName;int count;### Don't Use Single Letters (except in very limited scope)// Badpublic User GetU(int i) { }// Goodpublic User GetUser(int userId) { }// Acceptable in limited scopefor (int i = 0; i < count; i++) { }users.Select(u => u.Name); // Lambda parameter### Don't Encode Type in Name// BadList&lt;User&gt; userList;Dictionary&lt;int, User&gt; userDictionary;// GoodList&lt;User&gt; users;Dictionary&lt;int, User&gt; usersById;### Don't Use Ambiguous Names// Badvar data = GetData();void Process() { }void Handle() { }// Goodvar users = GetUsers();void ProcessOrder() { }void HandleUserCreatedEvent() { }## Quick Follow the same patterns throughout your codebase- **Clarity**: Names should be self-documenting- **Context**: Consider where the name is used- **Domain**: Use business language that stakeholders understand- **Standards**: Follow language and framework conventions- **Team Agreement**: Align with your team's preferences- **Tooling**: Configure linters to enforce conventions- **Documentation**: Keep naming guidelines accessible to the team
